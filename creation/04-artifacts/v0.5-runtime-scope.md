# V0.5 Runtime Scope — CreateOS Bootstrap

**Status**: Implemented (Primary runtime as of 2025-12-12)

---

## 1. Objectives

V0.5 transitions the CreateOS Bootstrap (C01) from the V0 dual-chat runtime (ChatGPT Project + Codex) to a **GitHub Copilot-based runtime** while preserving V0 as a fallback mode.

**Primary objectives**:

1. **Simplify session bootstrap** — Single-phrase `start createOS` boot using GitHub Action + JSON report.
2. **Architect-Developer separation** — Architect (Copilot Space) produces reasoning and work packages; Developer (Copilot Agent) executes PRs.
3. **Deterministic, auditable operations** — All changes tracked via Git, session_id tagging, and structured memory entries.
4. **Preserve V0 fallback** — V0 dual-chat + Git-backed tools remain functional for continuity and troubleshooting.

---

## 2. Roles (V0.5 Primary Runtime)

### Architect (Copilot Space)
- **Platform**: GitHub Copilot Space
- **Prompt**: `creation/04-artifacts/agent-prompts/architect-prompt.md` (full spec) and `architect-prompt-runtime.md` (concise runtime version)
- **Responsibilities**:
  - Validate repository state and rehydrate Creation context
  - Read `.createos/index.json`, memory, tasks, and progress logs
  - Produce deterministic work packages for Developer agent
  - Maintain auditable session history via `session_id` tagging

**Session boot protocol**:
- Human types: `start createOS`
- Architect instructs founder to run GitHub Action (`CreateOS – Start Session`)
- On completion, founder types: `boot report ready`
- Architect reads `creation/04-artifacts/boot/LATEST.json` and emits readiness summary

### Developer (Copilot Agent)
- **Platform**: GitHub Copilot Agent
- **Prompt**: `creation/04-artifacts/agent-prompts/developer-prompt.md`
- **Responsibilities**:
  - Receive work packages from Architect
  - Implement changes as PRs with explicit diffs
  - Execute tests and verify acceptance criteria
  - Commit with `session_id` in commit message

## 3. V0.5 Smoke PR Acceptance Criteria

To validate V0.5 runtime end-to-end, the following must be demonstrated:

1. **Session bootstrap**:
   - Human runs GitHub Action (`CreateOS – Start Session`) on `main`
   - Action generates valid boot JSON with `session_id`, `head_sha`, and `suggested_next_actions`
   - Boot JSON auto-committed to `creation/04-artifacts/boot/LATEST.json`
   - Architect reads LATEST.json and produces readiness summary

2. **Readiness banner parity**:
   - Architect emits: `SESSION: <session_id> @ <head_sha>`
   - Architect displays last progress summary, 3–6 open tasks, and 1–3 next-step options
   - Output matches V0 readiness format

3. **Architect → Developer handoff**:
   - Architect produces a `SESSION HANDOFF – Developer Instructions` block
   - Block includes: `session_id`, `architect_summary`, `PR_TITLE`, `BRANCH`, `COMMIT_MESSAGE`, `FILES_AND_DIFFS`, `TESTS`, `ACCEPTANCE`, `AUDIT`
   - Developer agent receives block and opens PR

4. **Session_id tagging**:
   - Commit message includes: `<session_id>: <description>`
   - PR title or body references `session_id`

5. **CI index refresh**:
   - After PR merge, CI workflow (`refresh-index.yml`) runs successfully
   - `.createos/index.json` includes new artifacts and decision docs
   - No regressions in index schema or file paths

6. **Minimal doc-fix PR**:
   - End-to-end flow tested with a small documentation change (e.g., typo fix, README update)
   - PR merged cleanly, CI passes, session closes properly

## 4. Coexistence with V0 (Fallback Mode)

V0 dual-chat runtime remains **fully operational** and documented as a fallback:

- **V0 boot protocol**: See `creation/06-decisions/session-start-protocol.md` (Section 4)
- **V0 tools**:
  - `tools/close_session.py` — Session close + progress log update
  - `tools/add_memory_entry.py` — Manual memory append
  - `creation/08-progress/LATEST.json` — Warm-start fast path (future)
- **V0 use cases**:
  - GitHub Copilot unavailable or degraded
  - Emergency troubleshooting requiring manual control
  - Historical reference for V0 design patterns

**No new V0 features** should be added; V0 is maintenance-only. Tasks T002 and T003 remain `pending` with `mode: "maintenance"` to preserve fallback documentation.
